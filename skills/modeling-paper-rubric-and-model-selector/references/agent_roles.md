# 数学建模多 Agent 协作角色

用途：当需要把完整建模任务拆成多个子任务、组织并行工作、或借鉴多智能体工作流时读取。此文件借鉴开源数学建模 Agent 的工程化思路，但保持为通用角色规范。

## 推荐角色

### 1. Problem Analyst

- 输入：赛题 PDF/Word、附件说明、用户补充约束。
- 输出：逐问任务拆解、输入/输出、评价指标、边界条件、外部数据需求。
- 门禁：每一问必须有可量化输出和可验证方式。

### 2. Data Engineer

- 输入：附件数据、外部数据、字段说明。
- 输出：数据字典、清洗规则、缺失/异常统计、清洗后数据、可复现脚本。
- 门禁：不修改原始数据；所有衍生数据写入 `paper_output/data_cleaned/`。

### 3. Modeling Specialist

- 输入：任务拆解、数据质量报告、约束条件。
- 输出：基线模型、改进模型、目标函数、约束、求解方法、参数设置。
- 门禁：必须有基线；复杂模型必须说明为什么必要。

### 4. Experiment Runner

- 输入：模型方案、清洗数据。
- 输出：实验结果、指标表、对照实验、消融或敏感性分析。
- 门禁：结果必须可复现，图表写入 `paper_output/figures/`。

### 5. Paper Writer

- 输入：任务拆解、模型方案、实验结果、评分点对齐表。
- 输出：摘要、正文、结论、附录、参考文献。
- 门禁：每一问有明确结论；图表编号与正文引用一致。

### 6. QA Auditor

- 输入：题面、模型路线、数据报告、论文稿。
- 输出：PASS/FAIL、失败清单、修复建议。
- 门禁：失败项修复前不得合并最终稿。

## 协作顺序

1. Problem Analyst 先输出 A/B/C/D。
2. Data Engineer 根据附件和外部数据需求生成数据质量报告。
3. Modeling Specialist 选基线和改进路线。
4. Experiment Runner 产出指标、图表和对照实验。
5. QA Auditor 做阶段审计，通过后才进入 Paper Writer。
6. Paper Writer 生成微单元和合并稿。
7. QA Auditor 最终审计 `final_paper.md` 与 `final_paper.docx`。

## 冲突处理

- 题目约束与模型便利性冲突时，以题目约束为准。
- 数据不足与模型复杂度冲突时，优先退回可解释基线和外部数据补充。
- 写作篇幅与事实证据冲突时，优先保证证据链，不用空泛段落掩盖缺失实验。
- 多角色结论冲突时，由 QA Auditor 列出冲突位置，再由 Modeling Specialist 或 Data Engineer 修复。
